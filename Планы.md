# Планы перед продом

## Сделано

- Passwordless auth-service добавлен отдельным сервисом: email magic-link, роли `player`/`author`/`admin`, access/refresh cookies, logout, logout-all, audit, dev magic-link endpoint.
- Основной API интегрирован с auth через JWT/cookie filter; author/admin ручки больше не должны опираться на `dev-admin-token`.
- PostgreSQL заменил H2 в runtime для `api` и `auth-service`: отдельные compose-сервисы, volumes, Flyway, PostgreSQL driver, индексы, backup/restore в README.
- H2 оставлен только для in-memory тестов.
- Игровые сохранения и каталог игрока реализованы на базе `game_sessions`: есть `save_name`, список всех прохождений игрока, фильтр по истории, продолжение сессии, переименование слота, кнопки `Продолжить`/`Новая игра`, понятные пустые состояния и сценарий "новая игра не затирает прогресс". Сессии защищены по текущему `user.id`.
- Публикационный workflow частично усилен: есть версии/snapshots, статусы и архивирование.
- Базовая загрузка ассетов уже есть: upload path, public path, запись URL/metadata в БД.

## Активный приоритет

1. **Prod-секреты и prod-профиль**

   Убрать все dev defaults из production-запуска: обязательные `AUTH_JWT_SECRET`, DB passwords, SMTP secrets, cookie secure defaults, `AUTH_DEV_MODE=false`, запрет dev endpoints в prod. Разделить `application-prod.properties` для `api` и `auth-service`, добавить явные проверки старта при слабых секретах.

2. **SMTP и реальный email flow**

   Довести passwordless-вход до production: SMTP/env-конфиг, нормальный email template, `AUTH_PUBLIC_BASE_URL`, обработка ошибок доставки, логирование без утечки токенов, понятный UI при отправке/повторной отправке письма.

3. **Безопасность edge и CORS**

   Зафиксировать CORS allowlist без широких dev-origin в prod, security headers, cookie policy, CSRF-подход для cookie-auth, rate limiting на `/auth/login-link`, `/auth/verify`, import/publish/upload. Проверить, что nginx не прокидывает лишние dev endpoints наружу в prod.

4. **Тесты auth и ролей**

   Добавить integration-тесты для magic-link: одноразовость, TTL, replay, refresh/logout/revoke, blocked user, роли `author`/`admin`, запрет author/admin API без роли. Отдельно проверить PostgreSQL-friendly JDBC timestamps, чтобы не повторилась ошибка с `Instant`.

5. **E2E smoke перед релизом**

   Playwright-сценарий: пользователь получает dev magic link, входит, играет, открывает сохранения; автор открывает builder, импортирует историю, публикует; админ открывает `/auth/admin`, видит пользователей/истории и меняет роли. Запускать в CI на docker compose.

6. **Миграция старых H2 данных**

   Если на старой H2 базе есть нужные данные, подготовить одноразовый export/import: stories, scenes, choices, assets metadata, players, sessions, story_versions. Описать порядок: backup H2, поднять Postgres, импортировать, проверить counts и published stories.

7. **Права автора и модерация**

   Закрыть оставшиеся ownership-дыры: нельзя перезаписать чужой `story_key`, slug-конфликты должны быть понятными, удаление/архивирование только владельцем или admin, admin moderation flow для блокировки/архивации спорных историй.

8. **Публикационный workflow историй**

   Довести статусы до понятной модели `draft -> review -> published -> archived`, добавить предпросмотр неопубликованной истории, откат к версии, заметки к публикации, аудит действий автора/admin.

9. **Ассеты: хранение и уборка**

   Сейчас есть локальный upload. Для прода нужен выбор стратегии: managed object storage/S3/MinIO или отдельный volume с backup. Добавить MIME sniffing, лимиты по типам, preview, удаление неиспользуемых файлов после переимпорта истории, защиту от path traversal и слишком больших загрузок.

10. **Удаление слотов сохранений**

   Добавить удаление отдельного прохождения: `DELETE /api/sessions/{sessionId}`, проверку владельца по текущему `user.id`, кнопку удаления в UI и тест, что чужую сессию удалить нельзя.

11. **Валидация Story JSON**

   Усилить проверки: недостижимые сцены, циклы без выхода, пустые тексты, битые asset URL, несовместимые типы переменных, дубли `key/slug`, неиспользуемые ассеты, слишком большой JSON.

12. **Наблюдаемость**

   Request id, структурные логи, метрики ошибок/латентности, `/actuator/prometheus`, отдельные DB health indicators, алерты на 5xx, auth failures, рост очереди/latency, заполнение upload volume.

13. **CI/CD**

   GitHub Actions: `gradlew test`, frontend unit tests, Playwright e2e, Docker build, compose smoke-test, публикация образов. Минимум: блокировать merge при красных тестах.

14. **Документация**

   README частично устарел и местами битый по кодировке. Привести в UTF-8, обновить Story JSON contract, auth flow, роли, PostgreSQL backup/restore, SMTP, prod env, локальный dev flow.

15. **Домен, TLS и reverse proxy**

   Настроить production reverse proxy: HTTPS, gzip/brotli, cache policy для статических файлов, отдельный домен или защищенный path для builder, redirect HTTP->HTTPS.

## Зафиксированные решения для прода

- Домен куплен: `fraerapp.ru`. Production public base URL: `https://fraerapp.ru`; для magic-link писем использовать `AUTH_PUBLIC_BASE_URL=https://fraerapp.ru`.
- SMTP поднимаем свой для `noreply@fraerapp.ru`: опциональный `compose.mail.yaml` с `docker-mailserver`, `mailserver.env` хранится локально и не коммитится. Перед публичным входом обязательно настроить DNS: A/MX/SPF/DKIM/DMARC и reverse DNS/PTR у провайдера IP, иначе magic-link письма могут попадать в спам.
- Все реальные секреты храним в локальном env-файле рядом с проектом на машине, где запускается prod. Файл с секретами не коммитим, добавляем в `.gitignore`; в репозиторий кладём только `.env.example` без настоящих значений.
- Production можно поднять на домашнем ПК со статическим IP `92.100.70.96`, запуск через Docker Compose. DNS `fraerapp.ru` должен иметь A-запись на `92.100.70.96`.
- Builder доступен только пользователям с ролью `author` или `admin`.
- Регистрация публичная через email magic-link.
- Первый admin email: `culibiaka2012@yandex.ru`.

## Промт для следующего шага: prod-секреты и prod-профиль

Реализуй production-ready конфигурацию FraerApp после перехода на PostgreSQL и passwordless auth. Нужно убрать dev defaults из production-запуска, разделить `application-prod.properties` для `api` и `auth-service`, добавить обязательные env-переменные и fail-fast проверки для `AUTH_JWT_SECRET`, DB credentials, SMTP credentials, `AUTH_PUBLIC_BASE_URL`, CORS allowlist и cookie policy. В prod `AUTH_DEV_MODE` должен быть `false`, dev endpoints не должны быть доступны, cookies должны быть secure, а слабые/дефолтные секреты должны останавливать старт сервиса с понятной ошибкой. Обнови `compose.yaml`/`.env.example` так, чтобы локальный dev остался удобным, но production-профиль требовал явных секретов. Добавь тесты или smoke-проверки на запрет dev defaults и обнови README.

## Промт для шага: auth и role tests

Добавь integration-тесты для auth-service и основного API: magic-link создается и используется один раз, истекает по TTL, replay запрещен, refresh token ротируется, logout и logout-all отзывают сессии, blocked user не может пользоваться токенами, роли `author` и `admin` реально защищают ручки, пользователь без роли получает 403. Проверь PostgreSQL-compatible работу timestamp-полей через `JdbcTemplate`, чтобы сохранение `created_at`, `expires_at`, `used_at`, `revoked_at`, `blocked_at` не ломалось на Postgres.
