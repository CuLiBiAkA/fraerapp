# Планы перед продом

## Промт для пункта 1

Реализуй нормальную авторизацию и роли для FraerApp как отдельный backend-сервис авторизации, который станет единым источником пользователей, email-идентичности, ролей, сессий и токенов. Сейчас игрок определяется через `X-Player-Id`, а админские ручки защищены `dev-admin-token`; для продакшена нужно заменить это на passwordless-вход по ссылке из email: пользователь вводит email, auth-сервис создает одноразовый magic-link токен с коротким TTL, отправляет письмо, подтверждает переход по ссылке, создает server-side сессию и выдает приложению access/refresh токены или secure httpOnly cookies. После входа API FraerApp должен получать проверенную identity с ролями `player`, `author`, `admin`, а все player/author/admin ручки должны работать только через новую авторизацию.

Сделай реализацию под ключ: отдельный сервис в `docker compose`, собственную БД или отдельную схему Postgres, таблицы `users`, `email_login_tokens`, `sessions`, `refresh_tokens`, `roles`/`user_roles`, миграции, конфиги секретов, TTL, ротацию refresh-токенов, logout, revoke all sessions, защиту от replay magic-link токена, rate limiting на запросы входа и подтверждения, нормальные ошибки API и аудит событий входа. Предусмотри dev-режим без реальной отправки писем: magic link должен логироваться или отдаваться только в безопасном dev endpoint, а prod-режим должен отправлять письма через SMTP/env-конфиг. Email храни нормализованным, подтвержденным после первого входа, без паролей.

Интегрируй auth-сервис с текущим backend FraerApp: добавь middleware/filter проверки access token или сессии, убери доверие к клиентскому `X-Player-Id`, свяжи существующие `game_sessions`, авторские истории и статистику с новым `user.id`, добавь стратегию миграции старых anonymous/player-id данных, замени `dev-admin-token` на роль `admin`, а роль `author` используй для конструктора и публикации историй. Добавь endpoints `POST /auth/login-link`, `GET/POST /auth/verify`, `POST /auth/refresh`, `GET /auth/me`, `POST /auth/logout`, `POST /auth/logout-all`, админское назначение ролей и внутренний endpoint/JWKS или shared-secret механизм для проверки токенов основным API.

Обнови frontend: экран входа по email, состояние “письмо отправлено”, обработку callback-ссылки, хранение сессии через cookie или token storage согласно выбранной схеме, отображение текущего пользователя, logout, защиту авторских/admin экранов, понятные русские ошибки и редирект обратно на страницу, с которой пользователь начал вход. Добавь тесты: unit/integration для генерации и одноразового использования magic link, TTL, refresh/logout/revoke, проверки ролей, запрета доступа без роли, миграции от `X-Player-Id`, e2e-сценарий “пользователь получает dev magic link, входит, играет, автор открывает конструктор, админ видит защищенную ручку”. Документацию обнови по env-переменным, SMTP, ролям, локальному dev flow и продовой схеме безопасности.

## Промт для пункта 6

Реализуй игровые сохранения для FraerApp на базе существующих `game_sessions`. Сейчас выбор истории всегда создает новую сессию и может потерять удобный путь к текущему прохождению; нужно сделать сессии полноценными слотами сохранения: хранить имя слота, отдавать список прохождений игрока через API, позволять продолжить конкретную сессию, показывать в каталоге последнюю сессию истории с кнопками `Продолжить` и `Новая игра`, а также вывести общий список сохранений игрока. Новая игра должна создавать отдельный слот, существующий прогресс не должен затираться. Добавь миграцию БД, backend DTO/endpoints, UI-состояния и тесты на список/продолжение сохранений.

1. **Нормальная авторизация и роли**
   Сейчас авторство завязано на `X-Player-Id`, а админка на `dev-admin-token`. Для прода нужен login/password или OAuth, server-side sessions/JWT, роли `player`, `author`, `admin`, защита всех author/admin ручек.

2. **PostgreSQL вместо H2**
   H2 хорош для MVP, но прод лучше вести на Postgres: отдельный volume/managed DB, backup/restore, миграции Flyway, индексы под каталоги, сессии и авторские истории.

3. **Секреты и prod-конфиги**
   Убрать дефолтный `dev-admin-token`, выключить H2 console, разделить `application-prod.properties`, добавить обязательные env-переменные, CORS allowlist, security headers.

4. **Публикационный workflow историй**
   Сейчас импорт/публикация довольно прямолинейные. Добавить статусы `draft -> review -> published -> archived`, предпросмотр неопубликованной истории, версионирование и возможность откатиться.

5. **Загрузка ассетов**
   Сейчас ассеты это URL. Для прода нужен upload изображений/звука, ограничения размера/типа, хранение в S3/MinIO, превью, удаление неиспользуемых ассетов.

6. **Игровые сохранения**
   Сейчас создание истории стартует новую сессию. Нужны “продолжить”, список прохождений игрока, явное сохранение/слоты, защита от потери прогресса.

7. **Нормальная модерация и права автора**
   Уже есть ownership, но нужны: нельзя перезаписать чужой `key`, публичные slug-конфликты, удаление/архивирование только владельцем, админская модерация.

8. **UI-полировка конструктора**
   Добавить понятные русские тексты вместо части английских `Edit/Stats/New story`, пустые состояния, подтверждения удаления/очистки draft, индикаторы сохранения и ошибок API.

9. **Валидация JSON глубже**
   Проверять недостижимые сцены, циклы без выхода, пустые тексты, битые URL ассетов, переменные с несовместимыми типами, дубли `key/slug`, неиспользуемые ассеты.

10. **Набор e2e-тестов**
    Playwright: автор логинится, создаёт историю, ставит `showInStats`, публикует, игрок проходит, видит статы, возвращается в каталог. Это сильно снизит риск ломать MVP.

11. **Наблюдаемость**
    Логи запросов, request id, метрики по ошибкам/латентности, `/actuator/prometheus`, алерты, отдельные healthchecks для DB и edge.

12. **Rate limiting и anti-abuse**
    Сейчас есть nginx limit на `/api/`, но нужны лимиты на login/import/publish, максимальный размер JSON, защита от слишком больших историй и частых сессий.

13. **Документация Story JSON обновить**
    README уже частично битый по кодировке и не описывает `showInStats`, author workflow, новый catalog progress. Нужно привести в UTF-8 и обновить контракт.

14. **CI/CD**
    GitHub Actions или аналог: `gradlew test`, `node scripts/run-ui-tests.mjs`, Docker build, smoke-тест контейнеров, публикация образов.

15. **Домен, TLS, reverse proxy**
    Продовый nginx/Caddy/Traefik с HTTPS, gzip/brotli, cache policy для статических файлов, отдельные домены для игры и конструктора или защищённый builder path.

## Промт для пункта 5

Реализуй загрузку ассетов для авторского конструктора FraerApp. Сейчас ассеты задаются только как URL в Story JSON; нужно добавить авторский multipart upload изображений и звука с проверкой размера и MIME-типа, сохранением файлов в настраиваемое локальное хранилище, публичной раздачей по `/uploads/**`, записью ассета в историю как `{ id, type, url, metadata }`, обновлением UI конструктора через кнопку выбора файла у ассета, автоматической очисткой неиспользуемых загруженных файлов после переимпорта истории и тестом авторского upload workflow.
